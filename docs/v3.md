# Quick Send v3

## Decision Summary

Quick Send v3 is a cloud-backed web app for sending text and images across a
single user's devices. It replaces the v2 LAN room model with an account-based
device inbox.

The product source of truth is Cloudflare. A device is a business object under
an authenticated account, not a room member, peer, or Better Auth session.

## Product Scope

v3 MVP supports:

- Account login with Better Auth.
- Multiple devices under one account.
- Source-device conversations.
- Text messages.
- Single image messages.
- Private image storage in Cloudflare R2.
- Message and image retention for 30 days.
- Short polling for sync.

v3 MVP does not support:

- Rooms, groups, group passwords, join requests, or pairing.
- LAN-only operation.
- Native apps, Tauri, Expo, or mobile app packages.
- WebRTC transfer.
- Arbitrary files, video, PDF, zip, or directories.
- Migration from existing local SQLite room data.
- Socket.IO, WebSocket, SSE, or Durable Object fanout.

## User Model

Better Auth owns users, accounts, and sessions.

Quick Send owns devices.

After login, the browser creates or restores a local `deviceId`. The API stores
only a hash of the device id. Sessions may expire or be cleared without deleting
the device record.

Each device has:

- `id`
- `user_id`
- `device_id_hash`
- `display_name`
- `kind`
- `created_at`
- `last_seen_at`
- `revoked_at`

Device names are part of the core navigation. On first use after login, the user
must confirm the current device name. Names are unique per account, case
insensitive, and 2-32 characters long. The app may generate names from user
agent data, then append a suffix for duplicates.

Revoked devices cannot send new messages. Their history remains visible.

## Conversation Model

There are no bidirectional device-to-device conversations.

A conversation is a source device timeline:

- The left navigation lists devices under the current account.
- Selecting a device shows messages sent by that source device.
- The current device's own messages appear in its own conversation.
- Sending from the current device writes to the current device conversation.
- Messages store `sender_device_id` and `sender_device_name_snapshot`.
- Device rename updates the conversation title, but old message rows keep the
  sender name snapshot.

## Date Tabs

Within a source-device conversation, messages are grouped by local date.

MVP behavior:

- Default tab is `Today` when it has messages, otherwise the latest available
  recent date.
- Show only the latest 7 natural days that contain messages.
- Use an `Earlier` tab for older messages.
- Do not render empty date tabs.
- Date grouping uses the user's client timezone, passed to the API.

## Message Model

Messages belong to one user account and one source device.

Kinds:

- `text`
- `image`

Text messages:

- Store body in D1.
- Maximum length should stay small enough for transfer utility usage.

Image messages:

- Support one image per message.
- Supported MIME types: `image/jpeg`, `image/png`, `image/webp`, `image/gif`.
- Initial max original size: 20 MB.
- Store original image and thumbnail in private R2 objects.
- Store metadata in D1.
- Show thumbnail inline in the conversation.
- Open original image through an authenticated URL flow.

Messages and image objects expire after 30 days. Users may manually delete a
message earlier. Delete and expiry remove visibility for every device under the
account.

## Storage And Privacy

D1 stores auth tables, devices, messages, and image metadata.

R2 stores image bytes. R2 buckets are private:

- No public bucket.
- No permanent public object URL.
- Message rows store object keys, not public URLs.
- The frontend asks the Worker for image access.
- The Worker verifies the session, account ownership, and message visibility.
- The Worker proxies image upload and download for MVP.
- Presigned browser upload/download URLs are deferred. They require R2 S3 API
  credentials and extra CORS handling, so they are not part of the first build.

## Image Upload Flow

Image bytes go through authenticated Worker handlers for MVP.

Flow:

1. The frontend validates MIME and size.
2. The frontend creates a thumbnail locally.
3. The frontend sends a multipart request to `POST /api/messages/image`.
4. The Worker verifies the session and current device.
5. The Worker validates MIME, size, and metadata.
6. The Worker writes original and thumbnail objects to private R2.
7. The Worker creates the message and image object rows in D1.

If R2 write succeeds but message creation fails, a cleanup cron can remove
orphaned R2 objects by age and missing metadata.

## Sync Model

MVP uses short polling.

- Current conversation polling interval: 3-5 seconds.
- Switching source device or date tab triggers an immediate fetch.
- Sending text uses optimistic UI and then confirms with the API response.
- Image messages appear after R2 upload and message creation complete.
- Device `last_seen_at` is updated through normal API calls or a lightweight
  heartbeat.

Realtime push is deferred.

## Cloudflare Architecture

v3 is fully cloud-hosted on Cloudflare:

- Frontend: Vite SPA with React and TypeScript.
- API: Cloudflare Worker with Hono and TypeScript.
- Auth: Better Auth.
- Database: Cloudflare D1.
- Object storage: Cloudflare R2.
- Cleanup: Worker Cron.

No local Bun server remains in the product runtime path.

Local development may still use Bun or npm for frontend tooling, Wrangler for
Worker development, and local D1/R2 mocks where practical.

## Frontend Stack

Use:

- React
- TypeScript
- Vite
- TanStack Router
- TanStack Query
- HeroUI
- Tailwind CSS v4
- Lucide icons where useful

Do not use:

- Next.js
- Tauri
- Expo
- Socket.IO
- Custom UI primitive library

The UI should stay compact and utility-focused. The first screen is the actual
device conversation app, not a marketing page.

## Backend Style

Use TypeScript throughout.

Use Drizzle for D1 schema, migrations, and queries.

Better Auth should use the official Drizzle adapter path. Business repositories
should also use Drizzle query builder. Raw SQL should be limited to uncommon
queries that are clearer as explicit SQL fragments.

API inputs and outputs should have explicit validation. Prefer the existing
project validation style if it remains in the v3 branch.

Business tables:

- `devices`
- `messages`
- `image_objects`

Better Auth owns the semantics of its required auth/session/account tables, but
the physical D1 schema and migrations live in Drizzle.

## Draft API

Auth endpoints are provided by Better Auth.

Business endpoints:

- `GET /api/bootstrap`
- `POST /api/devices/current`
- `PATCH /api/devices/:deviceId`
- `POST /api/devices/:deviceId/revoke`
- `GET /api/devices`
- `GET /api/messages?sourceDeviceId=...&localDate=...&timezone=...`
- `GET /api/messages?sourceDeviceId=...&cursor=...&earlier=1`
- `POST /api/messages`
- `DELETE /api/messages/:messageId`
- `POST /api/messages/image`
- `GET /api/images/:imageId`

## Retention

Default retention is 30 days for text, image metadata, original images, and
thumbnails.

Worker Cron should:

- Delete expired message rows.
- Delete expired image metadata rows.
- Delete corresponding R2 objects.
- Delete old orphaned R2 uploads that never became messages.
